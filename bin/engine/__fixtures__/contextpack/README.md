# Context-pack golden fixtures

Fixtures for `bin/engine/contextpack.golden.test.mjs` (forge#2703) — each
subdirectory is one scenario exercised end-to-end through the full
`mineContext()` -> `assemblePack()` -> `validateContextPack()` pipeline
(`bin/engine/contextpack.mjs`, `packages/protocol/src/contextpack-schema.js`).

This is distinct from `bin/engine/contextpack.mine.test.mjs`, which
unit-tests `mineContext()` in isolation with inline fixtures and never runs
the result through `assemblePack()`/`validateContextPack()`.

## Fixture shape

Each scenario directory holds exactly two files:

- **`fixture.json`** — the raw mocked `gh` responses plus the parameters
  passed to `mineContext()`/`assemblePack()`:
  ```json
  {
    "description": "human-readable summary of what this scenario exercises",
    "issueNumber": 3001,
    "repo": null,
    "phaseId": "investigate",
    "assembleOpts": {},
    "issueView": { "number": 3001, "title": "...", "body": "...", "labels": [], "state": "OPEN", "milestone": null },
    "commentsRaw": "[...]",
    "timelineRaw": "[]",
    "prListById": {},
    "gistById": {}
  }
  ```
- **`expected-pack.json`** — the committed golden output of
  `assemblePack(phaseId, minedData)` for this fixture. **Never hand-typed.**
  Always generated by actually running the real pipeline once and writing
  its output — see "Regenerating a fixture" below.

## The POST-jq shape rule (forge#2717)

`mineContext()`'s `io.gh(args)` injection stands in for `gh` **and** its
`--jq` filter together — the mock never re-runs `jq` on what it returns. Every
field in `fixture.json` MUST already be in the shape the code reads *after*
its `--jq` filter runs, not GitHub's raw REST API shape:

| gh call | Fixture field | Shape |
|---|---|---|
| `issue view ... --json ...` | `issueView` | Native `--json` shape (no jq): `{number, title, body, labels, state, milestone}` |
| `api .../comments --paginate --jq '[...]'` | `commentsRaw` | POST-jq: array of `{id, author, body, createdAt}` — **not** `{id, user: {login}, body, created_at}` |
| `api .../timeline --paginate --jq '[...]'` | `timelineRaw` | POST-jq: plain array of PR numbers, e.g. `[42, 57]` |
| `issue list --label review-finding ... --json number,title` | `prListById["<pr>"]` | Native `--json` shape (no jq transform on this call): array of `{number, title}` |
| `gist view <id> --raw` | `gistById["<id>"]` | Raw string content, or `{"error": "message"}` to make the mock throw (simulates a 404/deleted gist) |

A fixture built against the wrong (pre-jq) shape will silently produce
`author: ""`/`createdAt: null` or similar — the exact defect class forge#2717
found and fixed in `contextpack.mine.test.mjs`. Always re-derive the shape
from the actual `--jq` filter in `bin/engine/contextpack.mjs`, don't guess
from GitHub's REST API docs.

### Pagination fixtures

`gh api --paginate` runs the `--jq` filter once **per page** and concatenates
each page's output — for an array-producing filter that is **one JSON array
literal per page, back-to-back on stdout**, not one combined array (see
`extractJsonArrayPages()`'s docblock in `contextpack.mjs`). A fixture that
needs to exercise pagination (e.g. `paginated-comments`) must set
`commentsRaw` to two or more concatenated array-literal strings — a single
large `JSON.stringify(bigArray)` never exercises the multi-page code path,
even if the array itself has >100 entries.

## Regenerating a fixture

There is no long-lived generator script committed to the repo (the one used
to produce these fixtures was a scratch file, deleted before commit). To add
or regenerate a fixture:

1. Write a small throwaway `.mjs` script under `bin/engine/` (temporary —
   delete it before committing) that:
   - builds an `io.gh` mock from your fixture's raw response data (see
     `ioForFixture()` in `contextpack.golden.test.mjs` for the exact
     dispatch-by-command-prefix pattern to reuse),
   - calls `mineContext(issueNumber, { io, repo })`,
   - calls `assemblePack(phaseId, minedData)`,
   - writes `fixture.json` (your inputs) and
     `JSON.stringify(assemblePack(...), null, 2)` to `expected-pack.json`.
2. Run it once from the repo root: `node bin/engine/your-scratch-script.mjs`.
3. Delete the scratch script — only the two JSON files per scenario are
   committed.
4. Add a `describe`/`it` block to `contextpack.golden.test.mjs` asserting
   the scenario-specific behavior (not just "matches the golden file" —
   see the existing 6 scenarios for the pattern: assert on the *reason*
   the scenario exists, e.g. `sentinelState === 'interrupted'` for the
   malformed-annotation scenario, not only a blind deep-equal).
5. Run `node --test bin/engine/contextpack.golden.test.mjs` and confirm it
   passes.
6. **Prove the golden test actually catches drift**: temporarily rename a
   field in `assemblePack()`'s output (e.g. `content` -> `contentX` in
   `bin/engine/contextpack.mjs`), re-run the suite, confirm one or more
   golden tests fail, then revert the temporary edit. Do this whenever you
   add a new scenario — a golden test that never fails when the code
   actually drifts provides no coverage.

## Required scenarios (forge#2680)

| Directory | What it proves |
|---|---|
| `empty-issue/` | No comments, no labels, no annotations — pack is still minimal but schema-valid |
| `truncated-comments/` | A comment carries a GitHub truncation marker AND is large enough to force `assemblePack()` to truncate the slice; miner does not crash, pack sets `slice.truncated = true` |
| `malformed-annotation/` | One annotation missing its `:COMPLETE` sentinel (still parses, `sentinelState: 'interrupted'`), one with an unterminated opening tag (not recognized as an annotation at all); a well-formed sibling annotation still parses correctly |
| `deleted-gist/` | A `KNOWLEDGE_GIST` annotation references a gist that 404s; `mineContext()` marks it `available: false` with a non-empty `error`, pack stays schema-valid |
| `satellite-repo/` | Cross-repo (satellite) issue — every mocked `gh` call carries the satellite repo (`-R` args / `repos/{repo}/...` path), pack shape is unaffected |
| `paginated-comments/` | >100-comment thread split across 2 `--paginate` pages (60 + 45 = 105); miner pages through ALL comments (no silent truncation at page 1), pack still respects the size cap |
